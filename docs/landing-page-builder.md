# EcoMate Landing Page Builder — Technical Specification

## Overview
A landing/sales page builder integrated into the EcoMate e-commerce admin panel. Supports 2 modes: **Template** (pre-built sections with editable content) and **Custom Code** (user provides full HTML generated by any AI tool). Pages live at `/landing/[slug]` with zero main-app JavaScript — completely separate from the storefront.

---

## Architecture

### Backend (NestJS + Prisma + PostgreSQL)

**Model: LandingPage**
```prisma
model LandingPage {
  id           String    @id @default(uuid())
  title        String
  slug         String    @unique
  pageType     String    @default("template")  // "template" | "custom"
  templateId   String?                           // "clothing" (only template available)
  sections     Json?     @default("[]")          // Array of section configs for template mode
  customHtml   String?   @db.Text                // Full HTML for custom mode
  customCss    String?   @db.Text
  productIds   Json?     @default("[]")          // ["uuid1", "uuid2"]
  comboIds     Json?     @default("[]")
  trackingJson Json?     @default("{}")
  isActive     Boolean   @default(false)         // Published state
  isDraft      Boolean   @default(true)
  publishedAt  DateTime?
  createdAt    DateTime  @default(now())
  updatedAt    DateTime  @updatedAt
}
```

**API Endpoints (all prefixed with `/api/landing-pages`):**
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/` | admin | List all (paginated) |
| GET | `/:id` | admin | Get by ID |
| GET | `/published/:slug` | public | Get published page |
| GET | `/preview/:slug` | public | Get any page (draft) |
| POST | `/` | admin | Create |
| PUT | `/:id` | admin | Update |
| POST | `/:id/publish` | admin | Publish (isActive=true, isDraft=false) |
| POST | `/:id/unpublish` | admin | Unpublish |
| DELETE | `/:id` | admin | Delete |

**Error Handling:**
- Duplicate slug → `409 Conflict: "Slug already in use"` (Prisma P2002)
- Invalid slug format → `400 Bad Request` (regex: `^[a-z0-9-]+$`)
- Missing title/slug → `400 Bad Request`
- Generic errors → re-thrown, caught by GlobalExceptionFilter

**Auto-Sync on Startup:**
On production startup (`NODE_ENV=production`), `main.ts` runs:
1. `npx prisma migrate resolve --rolled-back "..."` — marks failed migration as rolled back
2. `npx prisma db push --accept-data-loss` — creates missing tables
3. Raw SQL fallback via PrismaClient: `CREATE TABLE IF NOT EXISTS "LandingPage"` — always succeeds regardless of Prisma CLI availability

### Admin Panel (React SPA + Vite + TanStack Router)

**Route:** `/admin/op/landing-pages`
**Sidebar:** Under "Operational" panel, entry named "Landing Pages"

**Feature: LandingPages (functional component)**
State management via TanStack Query mutations.

**UI components:**
1. **List table** — shows all pages with Title, Slug, Type (template/custom), Status (Published/Draft), Updated date, Actions
   - Pagination (10 per page, prev/next)
   - Actions per row: View (if published), Edit, Duplicate, Publish/Unpublish, Delete

2. **Create/Edit Dialog** — 3 tabs:
   - **General**: Title, Slug (auto-generated from title), Page Type (template/custom toggle cards), Template selector (only "Clothing Sales Page")
   - **Content**:
     - Template mode: Load default sections button → shows 6 section cards. Each section has form fields (Title, Subtitle, CTA Text, Product selector dropdown populated from page-level assigned products, Image URL). Section types: hero, features, featured-grid, checkout-form, trust-badges, cta-footer.
     - Custom mode: Textarea for HTML + "Prompt Reference" modal button
   - **Products**: Two searchable selectors
     - Product search: debounced 300ms, min 2 chars, calls `/api/products?search=...`, shows results as dropdown with name/image/price. Selected shown as removable badges.
     - Combo search: same pattern, calls `/api/combos?search=...`
     - Both have "Clear all" buttons

3. **Prompt Reference Modal** — Example prompt + tracking API docs + copy button

4. **Unsaved changes warning** — Confirm dialog before closing dirty form

5. **Confirm dialogs** for delete operations

**Key UX behaviors:**
- On edit: fetches product names from API to display human-readable names (not UUIDs)
- Product dropdowns close on outside click (mousedown handler)
- Slug format validated on save: `/^[a-z0-9-]+$/`
- Empty custom HTML triggers warning toast on save
- Dirty state tracking via `markDirty()` calls on title/slug change

### Storefront (Next.js 16 + Tailwind CSS 4)

**Route:** `/landing/[slug]`
**Layout:** Completely separate minimal layout — NO main app header/footer/JS bundle
**Rendering:** Dynamic (ƒ), ISR revalidate: 300s
**CSP Middleware:** Middleware applies strict CSP to `/landing/*`:
```
default-src 'self'
script-src 'self' 'unsafe-inline'
style-src 'self' 'unsafe-inline' https://fonts.googleapis.com https://cdn.tailwindcss.com
img-src 'self' data: blob: https: http:
font-src 'self' https://fonts.gstatic.com
connect-src 'self'
frame-src 'none'
object-src 'none'
base-uri 'self'
form-action 'self'
```

**Data Flow:**
1. Page route fetches published LP by slug via `/api/landing-pages/published/:slug`
2. Fetches assigned products from `/api/products?ids=...`
3. If `?preview=true` → uses `/api/landing-pages/preview/:slug` (returns drafts too)
4. Renders template sections OR custom HTML

**Template Mode Components:**
- `HeroSection` — gradient bg, title, subtitle, CTA button
- `FeaturesSection` — 2x3 grid of feature cards (editable via admin)
- `FeaturedGridSection` — product grid from assigned products, with "Add to Cart" buttons
- `CheckoutFormSection` — order form (name, phone, address, payment method), submits to `POST /api/orders` with guest info
- `TrustBadgesSection` — 4 badges (Free Delivery, COD, Returns, Secure)
- `CTASection` — gradient CTA banner

**Custom Code Mode:**
- Raw HTML rendered via `dangerouslySetInnerHTML`
- User's inline JavaScript allowed (for EcoMate.track() calls)
- External scripts blocked by CSP

**Order Creation Flow (template mode checkout):**
1. User fills name/phone/address/payment
2. Form POSTs to `/api/orders` with:
   - items (from assigned products)
   - guestName, guestPhone
   - shippingAddress
   - paymentOptionType + gatewayCode
3. On success → "Order Placed!" screen
4. On error → error message shown in form

### Tracking System

**Inline Script** injected in all landing pages:
- `window.EcoMate.track(eventName, data)` — fires once per event (deduplicated by event name)
- Posts to `/api/tracking/events` via `sendBeacon` or `fetch` with `keepalive`
- Each event gets a unique `eventId: eventName + timestamp`

**Events (fire-once, no duplicates):**
| Event | Trigger | Where |
|-------|---------|-------|
| `PageView` | Page load | Auto |
| `ViewContent` | Hero CTA click, CTA button click | Template buttons |
| `AddToCart` | Product grid "Add to Cart" | Template product cards |
| `Lead` | Checkout form submission | Template checkout form |
| `InitiateCheckout` | After successful order creation | Template checkout form |
| `Purchase` | On order status change (instant or validated) | Backend, per main settings |

**Dedup Logic:**
```js
var fired = {};
window.EcoMate.track = function(event, data) {
  if (fired[event]) return; // Never fires same event twice
  fired[event] = true;
  // POST to /api/tracking/events
};
```

**Backend Handling:** `POST /api/tracking/events` forwards to Meta Pixel and TikTok Pixel per existing storefront configuration.

## Known Limitations / Potential Gaps

1. **Section-level productId** uses dropdown from page-level assigned products only, not independent product search
2. **Custom HTML code editor** is a plain textarea — no syntax highlighting (Monaco heavy)
3. **Template sections order is fixed** — no drag-reorder
4. **No page-level tracking config** in admin UI — tracking auto-uses main storefront pixels
5. **No A/B testing** — single variant per slug
6. **No scheduled publishing** — no future publish date
7. **No analytics dashboard** for landing page performance
8. **Custom HTML SEO** — no metadata generation (intentional, these are ad pages)
9. **No webhook on order create** from landing pages (uses same endpoint as main storefront)
10. **No image upload in section editor** — image URLs must be manually entered
11. **No mobile preview** in admin — must use `/landing/[slug]?preview=true` on mobile
12. **Combo search API** (`/api/combos?search=...`) may not support `search` parameter — depends on combos module implementation
